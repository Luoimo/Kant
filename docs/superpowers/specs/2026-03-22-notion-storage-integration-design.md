# Notion Storage Integration Design

**Date:** 2026-03-22
**Scope:** `NoteAgent` + `ReadingPlanAgent` — replace local `.md` file storage with Notion Databases
**Status:** Approved

---

## Context

`NoteAgent` and `ReadingPlanAgent` currently persist output via `LocalNoteStorage` / `LocalPlanStorage` — simple `.md` files under `data/notes/` and `data/plans/`. The Storage layer is already abstracted behind a `Protocol`, making backend substitution minimal-impact.

**Goal:** When `NOTION_API_KEY` is configured, store notes and plans as pages in dedicated Notion Databases. Fall back to local files when the key is absent.

---

## Decisions

| Question | Decision |
|---|---|
| Integration method | Notion REST API via `notion-client` Python SDK (not MCP client, not LLM tool calls) |
| Storage mode | Config-driven factory: Notion if `NOTION_API_KEY` set, local otherwise |
| Protocol `save()` return type | Change `-> str` to `-> str \| None` in both Protocols; call sites already handle `None` via `storage_path_out or ""` guard |
| Runtime Notion `save()` failure | Caught inside `notes_node` / `plan_node` (not inside storage class); answer still delivered to user, `storage_path` set to `""` |
| Runtime Notion `load()` failure | Exception propagates; request fails with an error response — no silent fallback for edit/extend |
| Config validation | Validated at first instantiation of `NotionNoteStorage` / `NotionPlanStorage`; raises `ValueError` with clear message if `NOTION_NOTES_DB_ID` / `NOTION_PLANS_DB_ID` is empty |
| Notion structure | Two separate Databases (Notes DB + Plans DB) |
| Content format | Markdown is canonical; stored as both rendered Notion blocks (human display) and raw `code` block (round-trip fidelity) |
| `edit`/`extend` page behaviour | Always creates a new Notion page per save (same as local file behaviour); orphaned pages are a known trade-off accepted for MVP |
| Markdown → blocks converter | Minimal custom converter (headings H1–H3, bullets, paragraphs, bold); avoids `md2notion` maintenance risk |

## Notion Database Schemas

### Notes Database (`NOTION_NOTES_DB_ID`)

| Property | Type | Value |
|---|---|---|
| `Title` | title | `note_id` (e.g. `note_thread1_1742600000`), generated by `notes_node` |
| `BookTitle` | rich_text | book name from retrieved docs metadata |
| `Format` | select | `structured` / `summary` / `qa` / `timeline` |
| `CreatedAt` | date | ISO 8601 UTC string: `{"start": "2026-03-22T00:00:00.000Z"}` |
| `Tags` | multi_select | reserved, not populated initially |
| Page body | blocks | Markdown rendered as Notion blocks + raw `code` block at end |

### Plans Database (`NOTION_PLANS_DB_ID`)

| Property | Type | Value |
|---|---|---|
| `Title` | title | `plan_id` (e.g. `plan_thread1_1742600000`), generated by `plan_node` |
| `BookTitles` | rich_text | comma-separated book titles |
| `PlanType` | select | `single_deep` / `multi_theme` / `research` |
| `CreatedAt` | date | ISO 8601 UTC string: `{"start": "2026-03-22T00:00:00.000Z"}` |
| `ProgressSummary` | rich_text | e.g. `3 sections completed` |
| Page body | blocks | Markdown rendered as Notion blocks + raw `code` block at end |

`storage_path` in `GraphState` stores the Notion **page ID** (36-char UUID) returned by `pages.create()`.

`list()` returns a list of Notion page IDs (not filesystem paths). This is semantically consistent — callers pass the returned strings to `load()` — but differs from `LocalNoteStorage` which returns full paths. No existing call site in the graph nodes passes `list()` output directly to `load()`, so there is no breakage.

## Content Storage Strategy

Notion API does not accept raw Markdown — content must be converted to block objects. Round-trip fidelity is critical for `edit`/`extend` actions.

**Write (`save`):**
1. Convert Markdown → Notion blocks using a minimal custom converter (H1/H2/H3, bullet lists, numbered lists, bold, plain paragraphs)
2. Append one additional `code` block containing the raw Markdown string
3. Call `client.pages.create(parent=..., properties=..., children=first_100_blocks)`
4. If total blocks exceed 100, append remaining blocks in batches of 100 via `client.blocks.children.append()` — both the rendered blocks and the `code` block must respect this limit

**Read (`load`):**
1. Fetch all blocks via `client.blocks.children.list(block_id=page_id)`, following `has_more` cursor for pagination
2. Find the last `code` block — extract raw Markdown directly
3. If no `code` block found, fall back to concatenating `plain_text` from all blocks (degraded, display-only quality)

This ensures `load()` returns exactly what `save()` received, preventing format drift across `edit`/`extend` cycles.

## Architecture

### Implementation Order

The following steps must be completed in order. The Protocol files are **not yet updated** — this must happen before `NotionNoteStorage` is written, or static type checkers will flag a non-conformance.

1. Update Protocol `save()` signatures (step below)
2. Add try/except around `save()` in `notes_node` and `plan_node`
3. Implement `NotionNoteStorage` / `NotionPlanStorage`
4. Update `__init__.py` factory functions
5. Update `orchestrator_agent.py` call site

### Protocol Changes

**`backend/storage/note_storage.py`** and **`backend/storage/plan_storage.py`** (pending — not yet changed):

Change `save()` return type from `-> str` to `-> str | None` in both Protocol definitions and both Local implementations (Local implementations always return a path, so they naturally return `str`; the `| None` allows Notion failures to propagate as `None` without violating the Protocol).

### New Files

**`backend/storage/notion_storage.py`**

```
NotionNoteStorage
  __init__(settings)
    — validate NOTION_API_KEY, NOTION_NOTES_DB_ID not empty; raise ValueError if missing
    — init notion-client

  save(content, note_id) -> str | None
    — Convert Markdown → Notion blocks (custom converter)
    — Append raw code block
    — pages.create() with first ≤100 blocks
    — Append remaining blocks in batches via blocks.children.append()
    — Return page_id (str) on success
    — Does NOT catch exceptions here; caller (notes_node) wraps in try/except

  load(storage_path) -> str
    — blocks.children.list(block_id=page_id) with pagination
    — Extract raw Markdown from last code block
    — Raises on API failure (no silent fallback)

  list(prefix) -> list[str]
    — databases.query() with Notion title-type filter:
      {"property": "Title", "title": {"starts_with": prefix}}
    — Return [page_id, ...]

  delete(storage_path) -> None
    — pages.update(page_id, archived=True)

NotionPlanStorage
  — identical pattern, uses NOTION_PLANS_DB_ID
```

## Data Flow (new note, Notion enabled)

```
User request
  → notes_node
      → generates: note_id = f"note_{thread_id}_{timestamp}"
      → NoteAgent.run() → Markdown answer string
      → try: NotionNoteStorage.save(answer, note_id)
            → Markdown → Notion blocks (custom converter)
            → Append raw code block
            → pages.create(NOTES_DB_ID, {Title: note_id, ...}, blocks[:100])
            → blocks.children.append(...) for remaining blocks in batches
            → Returns page_id ("xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx")
        except: log, storage_path_out = None
      → GraphState.notes_last_output.storage_path = page_id or ""

Next edit/extend turn:
  → notes_node reads storage_path from notes_last_output
  → NoteAgent.run(action="edit", storage_path=page_id)
      → NotionNoteStorage.load(page_id)
            → blocks.children.list(page_id) with pagination
            → Extracts raw Markdown from code block
            → Returns original Markdown string
      → LLM modifies content
      → notes_node generates new note_id, calls save() → new Notion page created
      → Old page remains in DB (known trade-off, MVP)
```

## Error Handling

| Failure point | Behaviour |
|---|---|
| `save()` raises (network, rate limit, bad token) | Caught in `notes_node` / `plan_node` try/except; logged; `storage_path` set to `""`; answer still returned to user |
| `load()` raises (edit/extend path) | Exception propagates through `NoteAgent.run()` → `notes_node` → LangGraph framework → request fails with error response |
| `NOTION_API_KEY` set but `NOTION_NOTES_DB_ID` empty | `NotionNoteStorage.__init__()` raises `ValueError`; surfaces on first request that triggers storage instantiation |
| `NOTION_API_KEY` not set | Factory returns `LocalNoteStorage`; no Notion calls made |

## Known Limitations (MVP)

- **Orphaned pages:** Every `save()` creates a new Notion page, including for edit/extend operations. Old pages are not archived. Pages accumulate in the DB over time. A cleanup mechanism or update-in-place path can be added in a future iteration.
- **`load()` failure is fatal:** If Notion is unavailable during an edit/extend, the request fails. There is no local fallback for this path.
- **Converter coverage:** The custom Markdown → blocks converter covers common constructs only (H1–H3, bullets, numbered lists, bold, plain text). Complex Markdown (tables, nested lists, code fences) renders as plain paragraphs in Notion but is preserved losslessly via the raw `code` block.

## Out of Scope

- Bidirectional sync (Notion edits reflected back to app)
- Notion as RAG retrieval source
- Migration tool for existing local `.md` files
- Runtime fallback to local storage when Notion API is unavailable
- Update-in-place for edit/extend (avoids orphaned page accumulation)
